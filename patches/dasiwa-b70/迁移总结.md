# Dasiwa WAN2.2 B70/B60 Intel XPU migration summary

This document summarizes the migration result for the workflow package around:

- workflow: `DaSiWa-WAN2.2图生视频流-支持单图_双图_三图出视频json.json`
- delivery bundle: `patches/dasiwa-b70/`
- target runtime used in validation: **single Intel B60 / 24 GB effective budget**

It is written as a release-facing migration summary rather than a step-by-step lab notebook.

## 1. Workflow totals and functional scope

| Item | Value |
| --- | ---: |
| workflow nodes | `231` |
| links | `248` |
| final video outputs | `3` |
| structural/UI nodes | `110` |
| executable prompt nodes | `121` |
| executable nodes covered by full baseline | `105` |
| executable nodes covered by successful smoke only | `16` |
| executable nodes left unvalidated | `0` |

### What the workflow does

This workflow is not a single linear video path. It combines:

1. Wan image-to-video generation
2. first/last-frame guided generation
3. multi-frame reference generation
4. low-noise / high-noise LoRA-conditioned model paths
5. interpolation and MP4 output
6. prompt/VQA helper logic
7. VRAM/RAM cleanup and queueing helpers

So the migration had to preserve a **multi-branch workflow**, not just make one output node produce a file.

## 2. Migration result at a glance

### What succeeded

1. The original workflow graph was preserved; the source workflow JSON was not rewritten as part of the migration.
2. All three preserved output branches can be prompt-validated.
3. All three preserved output branches have successful reduced-resource smoke evidence.
4. All executable nodes are now accounted for across:
   - the authoritative converted prompt
   - full-run execution evidence
   - successful branch-smoke evidence
5. The remaining full-size blocker is no longer “unknown compatibility”; it is a measured **single-card memory limit** on the first full-size Wan denoise peak.

### What is still blocked

1. Full-size branch `54` at target fidelity still does not complete on a single 24 GB B60.
2. The original proprietary low-noise UNet filenames are still unresolved and are covered only by smoke-only compatibility aliases.
3. Some workflow helpers can be made compatible, but they are **not** currently meaningful Intel-XPU acceleration wins; they are compatibility/workaround paths.

## 4. Which nodes were effectively moved away from “GPU-first” behavior

These are the important cases where the workflow or custom-node ecosystem clearly assumed a CUDA/NVIDIA-style GPU path, but the Intel migration had to use **CPU placement**, **fallback behavior**, or **feature disablement**.

| Node / family | Intel migration result | Why it could not stay on the original GPU-oriented path |
| --- | --- | --- |
| `CLIPLoader` / text-conditioning path | kept CPU-biased | low compute density; preserving XPU headroom for Wan denoise is safer and usually better |
| `Qwen3_VQA` | CPU-biased with fallback behavior | upstream logic depended on CUDA capability checks, CUDA FP8 variants, bitsandbytes quantization, and CUDA cache APIs |
| `ImageResizeKJv2`, `LayerUtility: ImageScaleByAspectRatio V2` | CPU placement | Lanczos-style resize path was not treated as a safe Intel XPU path |
| `RIFE VFI` | not promoted as a trusted XPU-native optimization path | extra compute/memory stage with separate custom-node/kernel risk; validated functionally, not claimed as an Intel-XPU strength |
| `LaoLi_Lineup` | retained as a compatibility helper, not a CUDA memory-management path | original behavior relied on CUDA-specific memory statistics and synchronization semantics |
| `RAMCleanup`, `VRAMCleanup`, `easy cleanGpuUsed` | compatibility-cleanup only | original intent is GPU/CUDA cleanup, but Intel path must use device-agnostic model-management APIs |
| `PathchSageAttentionKJ` | disabled | SageAttention was not treated as the safe Intel XPU route in this migration |
| `ModelPatchTorchSettings.enable_fp16_accumulation` | forced off | original GPU-oriented accumulation setting was not kept for Intel XPU migration |

## 5. NVIDIA-oriented features that needed a gap, workaround, or disable

| NVIDIA/CUDA-oriented feature | Where it appeared | Intel-side handling |
| --- | --- | --- |
| `torch.cuda.get_device_capability()` | `Qwen3_VQA` | replaced with `comfy.model_management.should_use_bf16()` / device-aware logic |
| CUDA-only FP8 model variant selection | `Qwen3_VQA` `*-FP8` models | fallback to non-FP8 model on non-CUDA devices |
| bitsandbytes 4-bit / 8-bit quantization | `Qwen3_VQA` | disabled on non-CUDA devices; fallback to `none` |
| `flash_attention_2` as CUDA path | `Qwen3_VQA` | fallback to `sdpa` on non-CUDA devices |
| `torch.cuda.empty_cache()` / `torch.cuda.ipc_collect()` | `Qwen3_VQA`, cleanup helpers | replaced with `comfy.model_management.soft_empty_cache()` |
| `torch.cuda.synchronize()` | `LaoLi_Lineup`, Easy-Use cleanup | replaced with device-agnostic `mm.synchronize()` |
| `torch.cuda.memory_reserved()` / `torch.cuda.get_device_properties()` | `LaoLi_Lineup` | replaced with `mm.get_total_memory()` and `mm.get_free_memory()` |
| SageAttention-as-default path | `PathchSageAttentionKJ` | disabled in the Intel-safe runtime policy |
| original `enable_fp16_accumulation=true` policy | `ModelPatchTorchSettings` | overridden to `false` for Intel-safe runs |
| GPU-first Lanczos resize assumption | resize/aspect helper path | moved to CPU placement |

The key pattern is that the migration did **not** try to emulate CUDA internals. It replaced them with ComfyUI/model-management abstractions where possible, and moved unsupported functionality to CPU or safer attention/precision modes where needed.

## 6. Concrete code changes in this migration package

### 6.1 Main ComfyUI patch: `ComfyUI-main.patch`

This patch does not change the workflow JSON. It changes the tooling and runner behavior around it.

1. `workflow_asset_inventory.py`
   - adds package-id aliases so the workflow’s custom-node inventory resolves correctly instead of being left as `unknown`
   - examples: `comfyui_prompt_edit`, `comfyui_memory_cleanup`, `wan22fmlf`
2. `workflow_branch_runner.py`
   - creates the output directory before saving generated branch prompts
3. `workflow_to_prompt.py`
   - adds explicit conversion support for literal/widget-only nodes:
     - `Int`
     - `Float`
     - `String`
     - `KepStringLiteral`
     - `Checkpoint`
     - `Lora`
   - adds ordered-widget export support for:
     - `Prompt_Edit`
     - `LaoLi_Lineup`
     - `LoraLoaderModelOnly`
   - keeps these nodes from being silently dropped or half-exported during workflow -> API prompt conversion

### 6.2 Original branch fix: `ComfyUI-original-branch54-fix.patch`

1. normalizes `LoraLoaderModelOnly` selector values to basename-only form
2. prevents `/prompt` validation from pruning original branch `54` because of selector-name mismatch
3. adds unit coverage for the export rule

### 6.3 Custom-node patch: `ComfyUI-LaoLi-lineup.patch`

1. removes direct CUDA memory/property calls
2. replaces them with device-agnostic Comfy model-management APIs
3. keeps the node usable as a compatibility helper on Intel paths
4. does **not** prove that LaoLi is a strong Intel memory optimization; it only removes CUDA hard-coding

### 6.4 Custom-node patch: `ComfyUI_Qwen3-VL-Instruct.patch`

1. switches BF16 capability checks from CUDA-only logic to device-aware Comfy logic
2. falls back from CUDA-only FP8 model variants to non-FP8 variants on Intel/non-CUDA devices
3. disables bitsandbytes 4-bit / 8-bit quantization on non-CUDA devices
4. falls back from `flash_attention_2` to `sdpa` when CUDA is not available
5. replaces CUDA cache cleanup calls with device-agnostic cache cleanup
6. allows the VQA/helper branch to remain usable without pretending that CUDA-only Qwen paths exist on Intel

### 6.5 Custom-node patch: `ComfyUI-Easy-Use.patch`

1. removes raw `torch.cuda.synchronize()` dependency from cleanup flow
2. replaces it with device-agnostic synchronization via Comfy model-management APIs

### 6.6 Custom-node patch: `Comfyui_Prompt_Edit.patch`

1. hardens prompt-edit session handling when no live frontend client is attached
2. avoids getting stuck on UI interaction assumptions during backend or automated execution
3. preserves prompt-edit behavior in a non-interactive execution environment

## 7. What changed in workflow execution, even though the workflow JSON stayed intact

The migration intentionally preserved the original workflow file. The practical changes happened in the **converted prompt**, **device placement policy**, and **patched custom-node runtime behavior**.

### Runtime-policy changes applied around the workflow

1. `PathchSageAttentionKJ` is treated with an Intel-safe override policy rather than its original GPU-oriented default.
2. `ModelPatchTorchSettings.enable_fp16_accumulation` is treated with an Intel-safe override policy.
3. text encoders and helper branches are kept CPU-biased to preserve XPU memory headroom for the denoise path.
4. resize/aspect nodes are kept on CPU rather than assuming a GPU Lanczos path.
5. Qwen3 helper execution is kept on a safer non-CUDA path.

### Asset/workflow compatibility changes around the workflow

1. selector-backed model names needed normalization so prompt validation would not fail on basename-vs-path mismatches
2. proprietary low-noise UNets remain unresolved, so smoke-only compatibility aliases were used for smoke coverage
3. branch extraction and branch-only execution tooling were used so each branch could be validated without deleting or bypassing workflow nodes

## 8. Implemented gaps, workarounds, and their limits

| Gap | Workaround used | Limit of the workaround |
| --- | --- | --- |
| proprietary low-noise UNet source not available | smoke-only compatibility alias | proves graph execution only, not source-identical fidelity |
| CUDA-only FP8 / bitsandbytes Qwen path | non-FP8 / non-quantized fallback on Intel | keeps helper branch usable, but not feature-equivalent to CUDA path |
| CUDA memory-management helpers | swap to device-agnostic Comfy memory APIs | removes hard failure, but does not guarantee equal optimization benefit |
| GPU resize path not trusted on Intel | CPU placement | stable compatibility, but not an XPU acceleration win |
| full-size single-card denoise peak exceeds budget | reduced-resource smoke + documented blocked case | does not solve full-size production deployment on one B60 |

## 9. Final migration conclusion

This migration should be described as:

**graph-preserving migration success with complete executable-node coverage, plus documented Intel-XPU compatibility workarounds, but not full-size single-card production readiness.**

More plainly:

1. **The workflow is migrated enough to be audited, reproduced, and smoke-run end to end without dropping executable nodes.**
2. **The Intel package now explicitly captures the operator-level and feature-level gaps between NVIDIA/CUDA assumptions and Intel XPU reality.**
3. **The remaining hard blocker is mostly capacity and proprietary-asset fidelity, not an unexplained migration failure.**
