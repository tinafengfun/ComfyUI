## What changed during tuning

The tuning pass uncovered and fixed two generic harness issues before the path comparison was complete:

1. `workflow_perf_runner.py` now flattens nested `history.outputs` entry lists, which this workflow emits through `VHS_VideoCombine`.
2. `workflow_perf_runner.py` now keeps the report usable even if `ffprobe` is missing, recording file existence/size and marking the media probe as unavailable instead of aborting the whole run.

Because the first full baseline had already completed before the summarizer crash, its `report.json` was regenerated offline from the preserved `prompt.json`, `history.json`, and `xpu.csv`.

## Why the old tuning intuition failed here

The older `cartoon/Dasiwa-图生视频流.json` case was decode-bound and won by moving VAE decode back onto XPU. This workflow is different:

1. `VAEDecode` is tiny here (`2.24%` of node time).
2. The two sampler stages consume about `90%` of the total node duration.
3. Small loader-policy changes can only move the remaining `~10%` envelope unless they materially change sampler behavior.

## Full-run finalist decision

`P2` still had to be promoted once because it was the only non-regressive branch candidate. The full run rejected it:

- the completed baseline finished in `00:40:12`
- `F1-Full-IPEX` was still running at `00:47:59`
- at that point it had only produced:
  - `F1-Full-IPEX-54_00001.{mp4,png}`
  - `F1-Full-IPEX-131_00001.{mp4,png}`
- output `208` was still missing

That is enough evidence to classify the IPEX-enabled finalist as slower than the completed baseline, so it was stopped instead of burning more device time.
